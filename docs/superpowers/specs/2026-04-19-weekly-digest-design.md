# IHE Dashboard — Weekly Telegram Digest

**Date:** 2026-04-19
**Status:** Draft — pending review

## Goal

Send a Telegram message every **Monday at 7:00am ET** summarising business performance for **last week (Mon–Sun)** and **month-to-date**, across Shopify + Amazon sales, Klaviyo email, and Google + Meta ads (via GoMarble).

Complements the existing daily digest (`/api/digest`). Reuses the same Telegram bot, secret-gated endpoint, and GitHub Actions cron pattern.

## Delivery

- **Time:** 11:00 UTC every Monday (= 7:00am ET during EDT). Trade-off: during EST (Nov–Mar) it arrives at 6am ET. Accepted — one fixed UTC cron keeps it simple.
- **Trigger:** GitHub Actions cron calls `POST /api/digest/weekly?secret=<DIGEST_SECRET>`.
- **Channel:** Same Telegram bot + chat as daily digest. `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID` already in Render env.

## Scope

**In scope (v1):**
- Sales: Shopify + Amazon, last week + MTD, WoW % change on gross
- Top 1 product + top 3 states (last week only)
- Klaviyo: campaigns sent, open/click rate, attributed revenue, new subs, unsubs
- Ads via GoMarble: Google + Meta spend, conversions revenue, ROAS
- **AI analysis section** — 3–5 bullets of insight generated by Claude (Anthropic API) from the compiled numbers
- Single Telegram message (Markdown-free, plain text — same as daily)

**Out of scope (v2+):**
- Daily breakdown chart
- Per-campaign email detail
- Per-ad-creative breakdown
- Retention / repeat-purchase metrics
- eBay orders (not in dashboard data flow yet)

## Architecture

Three new modules + one new endpoint + one new workflow. Each module is importable and unit-testable in isolation.

### 1. `weekly_periods.py` — date math

```python
def last_week_range(now_et) -> tuple[datetime, datetime]:
    """Returns (start, end) for the most recently completed Mon–Sun week in ET."""

def mtd_range(now_et) -> tuple[datetime, datetime]:
    """Returns (start, end) for the current month through `now_et`."""

def prior_week_range(now_et) -> tuple[datetime, datetime]:
    """The week before last_week_range — used for WoW comparison."""
```

Pure functions. Extracted from `_compute_summary` to make them testable without hitting APIs.

### 2. `klaviyo_client.py` — email metrics

```python
def fetch_weekly_metrics(api_key: str, start: datetime, end: datetime) -> dict:
    """Returns {
      'campaigns_sent': int,
      'open_rate': float,       # 0-100
      'click_rate': float,
      'attributed_revenue': float,
      'new_subscribers': int,
      'unsubscribes': int,
    }"""
```

Uses Klaviyo REST API (`https://a.klaviyo.com/api/`) — no MCP at runtime. API key stored as `KLAVIYO_API_KEY` env var. Endpoints:

- `GET /api/campaigns` filtered by `send_time` — counts campaigns sent
- `GET /api/campaign-values-reports` — open/click/attributed revenue
- `GET /api/metric-aggregates/` with metric `Placed Order` attributed to Klaviyo — fallback for revenue
- `GET /api/events?filter=metric=Subscribed to List` — new subs
- `GET /api/events?filter=metric=Unsubscribed` — unsubs

Errors are caught per-field; returns zeros + appends to `errors` list on failure rather than crashing the whole digest.

### 3. `gomarble_client.py` — ads metrics via MCP-over-SSE

```python
def fetch_weekly_ads(api_key: str, start: datetime, end: datetime) -> dict:
    """Returns {
      'google': {'spend': float, 'revenue': float, 'conversions': int, 'roas': float},
      'meta':   {'spend': float, 'revenue': float, 'conversions': int, 'roas': float},
      'blended_roas': float,
    }"""
```

Uses the official `mcp` Python SDK as a client, connecting to `https://apps.gomarble.ai/mcp-api/sse` with `GOMARBLE_API_KEY` env var. Calls:

- `google_ads_run_gaql` — custom GAQL for date range → spend, conversions, conversion value
- `facebook_get_adaccount_insights` — with `time_range` for the week → spend, purchase value, purchases

Async under the hood; exposed as sync wrapper using `asyncio.run`. 30-second timeout per MCP tool call; on failure returns zeros + error list entry (same resilience pattern as Klaviyo).

**Note:** confirm GoMarble API key works from a non-Desktop MCP client as part of implementation. If it requires a Claude-Desktop-signed handshake, fall back to deferring ads to v2 per earlier plan.

### 4. `insights_client.py` — Claude-generated analysis

```python
def generate_insights(report: dict) -> list[str]:
    """Returns 3–5 bullet strings of analysis based on the compiled report.
    Empty list on error — analysis section collapses to `—` in that case."""
```

Calls Anthropic Messages API (`claude-sonnet-4-6` via the `anthropic` Python SDK) with a fixed system prompt that:
- Frames Claude as an e-commerce analyst for Iberian Ham Express
- Provides context: brand, top product, CA+FL focus, AOV, repeat-rate gap, 10-Pack bundle strategy
- Instructs: return exactly 3–5 bullets, each under 120 chars, each actionable or observational — no fluff
- Returns bullets as a JSON array for reliable parsing

Input is the compiled report dict (sales + Klaviyo + ads numbers for last week, prior week, MTD). Temperature 0.3 for consistency. 15-second timeout. Uses prompt caching on the system prompt (same framing every week → cache hit after first call).

API key in `ANTHROPIC_API_KEY` env var. On failure, returns `[]` and logs the error — digest still sends without the Analysis section rather than blocking.

### 5. `weekly_digest.py` — aggregator + formatter

```python
def build_weekly_report() -> dict:
    """Orchestrates all four sources + generates insights. Returns the full report dict."""

def format_weekly_message(report: dict) -> str:
    """Pure function. Takes report dict, returns Telegram message string."""
```

Uses `_compute_summary` for Shopify + Amazon (refactored to accept explicit `(start, end)` args so weekly can call it twice — once for last week, once for MTD — and once more for prior-week WoW). Calls `generate_insights(report)` last, after all numerical sources have resolved, so Claude has the full picture.

### 6. `POST /api/digest/weekly` endpoint (in `api_server.py`)

1. Secret check (same as `/api/digest`) → 401 / 503 behaviour identical.
2. Build report via `build_weekly_report()`.
3. Format via `format_weekly_message(report)`.
4. Send via existing `send_telegram(token, chat_id, text)`.
5. Return `{"ok": True, "message": message, "errors": report["errors"]}`.

### 7. `.github/workflows/weekly-digest.yml`

```yaml
name: Weekly Telegram Digest

on:
  schedule:
    - cron: '0 11 * * 1'      # 11:00 UTC Monday = 7am EDT / 6am EST
  workflow_dispatch:

jobs:
  digest:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl --fail -X POST \
            "https://ihe-dashboard.onrender.com/api/digest/weekly?secret=${{ secrets.DIGEST_SECRET }}"
```

Reuses `DIGEST_SECRET`. Nothing new to configure in GitHub.

## Error Handling

| Scenario | Behaviour |
|---|---|
| Klaviyo API fails | Email section shows `—`, error logged in response |
| GoMarble MCP fails or times out | Ads section shows `—`, error logged |
| Shopify/Amazon fails | Same as daily digest — partial data, error appended |
| Telegram API fails | 500 response → GitHub Actions run fails visibly |
| All four sources fail | Still sends a header-only message rather than silent failure |
| Anthropic API fails or times out | Analysis section shows `—`, numeric data still sent |
| Anthropic returns malformed JSON | Parse error logged, Analysis section shows `—` |
| Secret missing | 503 (not configured) |
| Wrong secret | 401 |
